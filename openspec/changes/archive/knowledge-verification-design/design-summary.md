# 隐含知识验证机制 - 设计总结

## 项目概述

**目标**：为POEMaster系统构建一个完整的隐含知识验证机制，确保推理过程中产生的隐含知识（非POB数据直接包含的知识）能够通过POB代码验证，从而提高知识库的可信度。

**核心问题**：
- 启发式推理会产生大量隐含知识
- 这些知识未经验证，可信度低
- 如果错误知识进入知识库，会污染推理结果

**解决方案**：
- 三层POB代码搜索验证
- 知识状态分类管理
- 自动验证 + 人工确认混合模式
- 完整的冲突检测和解决机制
- 知识演化和衰减管理

## 设计原则

### 1. 验证优先 (Verification First)
- 所有隐含知识必须经过验证
- 证据强度不足的保持pending状态
- 用户决策可以覆盖自动验证结果

### 2. 状态透明 (Status Transparency)
- 四种状态：verified(100%)、pending(50%)、hypothesis(30%)、rejected(0%)
- 每个状态有明确的转换条件和权限
- 状态影响知识在推理中的权重

### 3. 流畅交互 (Smooth Interaction)
- 自动验证不阻塞推理过程
- 用户可以批量处理待确认知识
- 冲突检测和处理自动化

### 4. 证据驱动 (Evidence-Driven)
- 所有决策基于证据链
- 证据强度量化（0-1）
- 多源证据组合评估

## 关键组件

### 1. POB代码搜索器 (POBCodeSearcher)

**三层搜索策略**：

| 层级 | 搜索类型 | 证据强度 | 示例 |
|------|---------|---------|------|
| Layer 1 | 显式stat定义 | 1.0 | StatDescriptions/*.lua中的stat定义 |
| Layer 2 | 代码逻辑 | 0.8 | CalcModules中的条件判断和计算逻辑 |
| Layer 3 | 语义推断 | 0.5 | 命名约定、共现关系、模式匹配 |

**搜索方法**：
- `search_stat_definition(stat_id)` - 搜索stat定义
- `search_skilltype_constraint(skill_type)` - 搜索类型约束
- `search_calc_logic(mechanism, entity)` - 搜索计算逻辑
- `semantic_inference(subject, predicate)` - 语义推断

### 2. 证据评估器 (EvidenceEvaluator)

**证据组合策略**：
```python
综合强度 = Σ (证据强度 × 权重) / Σ 权重

权重分配：
- 显式stat: 0.4
- 代码逻辑: 0.3
- 模式匹配: 0.2
- 语义推断: 0.1
```

**决策标准**：
- 强度 ≥ 0.8：自动验证
- 强度 0.5-0.8：推荐验证
- 强度 < 0.5：推荐拒绝

### 3. 冲突检测器 (ConflictDetector)

**冲突类型**：
1. **直接矛盾**：同一知识有两个矛盾值（最高优先级）
2. **推断冲突**：知识之间隐含矛盾（中优先级）
3. **证据冲突**：证据来源矛盾（低优先级）
4. **版本冲突**：不同版本的知识冲突（特殊处理）

**解决策略**：
```
证据仲裁 → 条件限定 → 状态降级 → 用户决策
```

### 4. 模式发现器 (PatternDiscoverer)

**模式类型**：
- **类型-属性模式**：`Spell + Fire => FireDamage`
- **因果模式**：`Trigger + Cast => EnergyCost`
- **绕过模式**：`SpecialStat => NoEnergy`

### 5. 知识演化管理器 (KnowledgeEvolutionManager)

**演化策略**：
| 变更比例 | 策略 | 说明 |
|---------|------|------|
| < 5% | 增量更新 | 只更新受影响的实体和知识 |
| 5-20% | 选择性更新 | 保留用户知识，重建系统知识 |
| > 20% | 完全重建 | 清空知识库，重新初始化 |

**知识衰减**：
- 未验证pending知识：30天半衰期衰减
- 无引用verified知识：标记为待重验证

## 知识状态系统

### 状态定义

| 状态 | 置信度 | 来源 | 权限 |
|------|--------|------|------|
| verified | 100% | POB数据直接包含或POB代码验证 | 所有推理 |
| pending | 50% | 启发式推理产生，待验证 | 受限推理 |
| hypothesis | 30% | 启发式假设，未经测试 | 仅探索推理 |
| rejected | 0% | 验证失败或用户拒绝 | 不参与推理 |

### 状态转换

```
pending ────[自动验证]────> verified
   │                            ▲
   │                            │
   └────[用户确认]──────────────┘
   │
   └────[证据不足]────> rejected
   
hypothesis ────[验证成功]──> pending
      │
      └────[验证失败]──> rejected
```

## 待确认知识角色

### 在不同推理阶段的作用

| 推理阶段 | 待确认知识角色 | 权重 | 说明 |
|---------|---------------|------|------|
| 查询推理 | 降权使用 | 0.5 | 作为补充结果 |
| 发现推理 | 启发线索 | 0.3 | 帮助发现新知识 |
| 验证推理 | 主动目标 | 1.0 | 优先验证pending知识 |
| 扩散推理 | 阻塞 | 0 | 不参与扩散 |
| 因果推理 | 弱证据 | 0.2 | 作为辅助证据 |
| 组合推理 | 阻塞 | 0 | 不参与组合 |

## 用户交互设计

### 核心界面

1. **待确认知识仪表板** - 概览、优先队列、批量处理
2. **知识详情卡片** - 证据链、自动验证结果、用户决策
3. **批量处理界面** - 筛选、选择、批量操作
4. **验证历史记录** - 决策追溯、报告生成

### CLI命令

```bash
# 查看待确认知识
python -m poe_data_miner verify list --status pending --top 10

# 验证单条知识
python -m poe_data_miner verify accept <id> --reason "POB代码验证"

# 批量处理
python -m poe_data_miner verify batch-auto --strategy smart

# 导出报告
python -m poe_data_miner verify export --format markdown
```

## 监控和日志

### 关键指标

1. **验证性能**：单次验证<1s，批量(10条)<5s
2. **自动验证率**：目标≥80%
3. **知识质量**：平均置信度≥0.8，覆盖率≥70%
4. **系统健康**：数据库完整性、POB数据新鲜度

### 告警规则

- 验证性能下降（P95>5s）
- 自动验证率过低（<70%）
- 待确认知识堆积（>20%）
- 知识冲突频发（>5个）
- POB数据过期（>30天）

## 测试策略

### 测试层次

1. **单元测试** - 核心逻辑函数（90%覆盖率）
2. **集成测试** - 模块间交互（80%覆盖率）
3. **系统测试** - API接口、性能（关键路径100%）
4. **E2E测试** - 完整用户场景（核心场景100%）

### 性能目标

- 单次验证：<1秒
- 批量验证(10条)：<5秒
- POB搜索缓存命中率：>70%
- 并发验证：支持5线程并发
