## Context

### 背景
POE数据矿工技能需要从POB的Lua文件中提取游戏数据，建立知识索引，支持智能问答。核心挑战：
- Lua文件分散，数据类型多样（实体、映射、计算代码）
- 游戏机制复杂，需要建立实体-规则-机制的关联
- 用户问题是发散式的，不能仅依赖预设路径

### 当前状态
- 技能目录已创建，包含基础的脚本结构
- 缺乏真实的数据提取和问答功能

### 约束
- 必须使用Python标准库，无第三方依赖
- SQLite嵌入式存储，无需额外服务
- 用户确认通过ask_followup_question实现

## Goals / Non-Goals

**Goals:**
- 实现完整的数据扫描和索引系统
- 建立三层规则提取机制（SkillStatMap、stats组合、计算代码）
- 实现链式索引 + 关联图的混合查询架构
- 支持增量学习和用户确认
- 支持版本更新后的知识恢复

**Non-Goals:**
- 不实现联网获取数据（社区知识、API）
- 不实现图形界面
- 不实现自动游戏测试

## Decisions

### 1. 存储架构：SQLite + YAML混合存储

**选择**: SQLite存储核心数据，YAML存储人类可读配置

**原因**:
- SQLite查询性能比YAML高10-60倍
- SQLite支持索引和递归CTE（图遍历）
- YAML保持人类可读性（配置、启发记录）

**备选方案**:
- 纯YAML：简单但性能差，图查询弱
- Neo4j图数据库：功能强但需要额外服务，打包困难

### 2. 规则提取：分层提取策略

**选择**: 三层提取，互补使用
- Layer 1: 从stats组合提取实体属性
- Layer 2: 从SkillStatMap提取属性关系
- Layer 3: 从计算代码提取条件规则和公式

**原因**:
- 每层用最适合的方式提取特定类型知识
- 交叉验证确保完整性
- 避免依赖不可靠的文本匹配

**备选方案**:
- 纯文本匹配：不可靠，表达不一致
- 纯代码分析：实现复杂，覆盖不全

### 3. 查询架构：链式索引 + 关联图

**选择**: 已知问题用链式索引，未知问题用关联图发散搜索

**原因**:
- 链式索引对已知路径高效
- 关联图支持发散探索
- 两者互补，满足不同场景

**备选方案**:
- 纯图结构：复杂度高，已知路径效率低
- 纯链式索引：无法处理发散问题

### 4. 增量学习：启发记录 + 用户确认

**选择**: 记录问题和发现，版本更新后用问题重新探索

**原因**:
- 问题比技术细节更稳定
- 用户确认确保知识准确性
- 启发记录作为恢复种子

### 5. 预置边：最小化策略

**选择**: 只预置10-20条真正无法自动提取的知识

**原因**:
- 大部分知识可以从数据自动提取
- 用户探索可以补充发现
- 预置边作为种子，不是主体

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| Lua代码分析可能不完整 | 分层提取互补，关键规则预置 |
| 关联图初始化可能遗漏边 | 用户探索补充，增量学习 |
| 版本更新后知识可能失效 | 启发记录重建，未确认列表提醒 |
| 用户确认负担 | 可选确认，不确认不影响基本使用 |
| SQLite数据库可能过大 | 只存储必要数据，定期清理低价值边 |

## Migration Plan

1. **Phase 1**: 数据扫描和实体索引
2. **Phase 2**: 规则提取和规则库
3. **Phase 3**: 关联图构建
4. **Phase 4**: 问答引擎
5. **Phase 5**: 增量学习和确认机制
6. **Phase 6**: 恢复机制
