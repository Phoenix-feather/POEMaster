## pob-build-analyzer v1.0.10 — removeNodes key 修复 + 珠宝诊断增强

### 核心发现：CalcSetup.lua 中 removeNodes 的两种 key 格式

**CalcSetup.lua 中 `override.removeNodes` 使用两种不同的 key**：

1. **第 734 行**（普通天赋移除）：
   ```lua
   if not override.removeNodes or not override.removeNodes[node] then
   ```
   key = **node 对象**

2. **第 1291 行**（GrantedPassive 天赋移除）：
   ```lua
   if node and (not override.removeNodes or not override.removeNodes[node.id]) then
   ```
   key = **整数 node.id**

**影响**：之前 `what_if_nodes()` 只设置 `[node]=true`，导致 GrantedPassive 天赋（Megalomaniac 的 "The Spring Hare"、"Savoured Blood" 等）无法通过 removeNodes 移除，DPS 始终显示 0% 变化。

### 修复清单

| 修复项 | 说明 | 文件 |
|--------|------|------|
| removeNodes 双 key | 同时设置 `[node]=true` + `[node.id]=true` | what_if.py |
| 珠宝物品移除 | selItemId=0 + spec.jewels[nodeId]=nil | what_if.py |
| GrantedPassive DPS | 额外通过 notableMap 查找+removeNodes 移除 granted 节点 | what_if.py |
| 新字段 granted_dps_pct | 仅 granted 节点的 DPS 贡献 | what_if.py |
| 新字段 dps_source | "item_mods" / "granted_passives" | what_if.py |
| __init__.py 文档 | diagnose_jewels() 返回类型更新 | __init__.py |

## pob-build-analyzer v1.0.11 — DPS 来源拆解 (DPS Source Breakdown)

### 新增功能

**`dps_breakdown()`**：将当前 DPS 的每个公式项拆解到具体来源。

- **两层粒度**：第一层按公式项分组（Base Damage / Damage INC / MORE / Speed / CritChance / CritMultiplier / Lucky），第二层逐条 mod 来源
- **Output-driven**：从 `output.*` 非零值判断活跃组件，跳过未激活部分
- **Label 可读化**：Lua 端建立三张查表（nodeNames / skillNames / itemNames），支持辅助宝石名称解析
- **category_summary**：每个公式项按来源类别（Tree/Item/Skill/Base/Config）汇总
- **Base Damage 完整拆解**：宝石基础 + added flat damage(Tabulate) + addedMult + baseMultiplier
- **集成到 `full_analysis()`**：step 8，返回 `dps_breakdown` 字段

### 关键技术

- 核心 API：`ModStore:Tabulate(modType, cfg, ...)` 返回 `[{value, mod}]` 对
- `Tabulate("MORE")` 的 `entry.value` 是百分比原值（如 39 = "39% more"），非乘数
- `damageStatsForTypes` 逻辑在 Lua 端复现（CalcOffence.lua:52-63）
- 辅助宝石通过 `mainSkill.supportList` → `grantedEffect.modSource/name` 获取可读名

### 运行时修复

| 修复项 | 说明 |
|--------|------|
| Item 负数 ID | Lua 模式从 `^(%d+)` 改为 `^(%-?%d+)`，支持 `-1` 等负数 itemId |
| 辅助宝石名称 | skillNames 扩展收集 supportList，解决 "Skill SupportSpellCascadePlayer" → "Spell Cascade" |

### 验证结果

- TotalDPS: 27060, 13 个公式项
- 活跃伤害类型：Lightning, Cold, Fire
- 天赋标签：Potent Incantation, All Natural, Careful Assassin 等
- 装备标签：Rapture Shard, Damnation Grip, Adonia's Ego 等
- 辅助宝石标签：Spell Cascade

### 修改的文件

- `what_if.py`: dps_breakdown() + 6 个辅助函数 + full_analysis() 集成 + Item/Skill label 修复
- `__init__.py`: dps_breakdown() API 方法 + full_analysis() 文档更新

## pob-build-analyzer v1.0.12 — 珠宝来源分类修复 (Jewel Classification Fix)

### 核心发现：POB 珠宝 mod 的双 source 格式

1. **直接珠宝 mods**（珠宝自身 stat）：`"Item:{itemId}:{name}"`，在 Item.lua:1760 设置
2. **珠宝半径效果 mods**（影响附近天赋）：`"Tree:{nodeId}"`，在 CalcSetup.lua:276 设置
3. **GrantedPassive 珠宝 mods**（如 Megalomaniac "Allocates X"）：`"Tree:{grantedNodeId}"`，其中 grantedNodeId 是**天赋节点自身 ID**（如 11774 = The Spring Hare），不是珠宝插槽 nodeId（21984）

### 修复

- 扩展 Lua 端 `jewelNodeMap`：遍历 `jItem.modList` 找 GrantedPassive 条目，通过 `tree.notableMap[gpName]` 查找 granted 节点 ID 并映射到珠宝名称
- 新增 `META|jewel_nodes` 输出行，Python 端构建 `jewel_node_ids: set`
- `_classify_source()` 新增 `jewel_node_ids` 参数，当 `Tree:{nodeId}` 的 nodeId 在 jewel 集合中时返回 `"Jewel"` 而非 `"Tree"`

### 验证

- TotalDPS: 27,060（不变）
- Lucky Hits 正确标记为 `[Jewel] Megalomaniac, Diamond → The Spring Hare`
- 完整 full_analysis() 6 个 section 全部生成

### 完整分析报告已输出

Comet 构筑完整 DPS 优化报告，包含：
- 15 个公式项完全拆解（每个 mod 逐条列出来源）
- 24 个灵敏度维度（14 个有效 + 10 个无影响）

## pob-build-analyzer v1.0.13 — INC 去重 + Item 部位标注

### 用户反馈

1. Rapture Shard / Chimeric Spark 是珠宝，Item 来源应标注部位
2. Lightning/Cold/Fire 三个 INC 几乎完全重复，应按 mod 类别分组

### INC/MORE 重构

旧逻辑：对每个活跃伤害类型分别 `Tabulate("INC", cfg, "Damage", "LightningDamage", "ElementalDamage")`，产生 3 组几乎相同的 formula items（15 个）。

新逻辑：按 mod 名称类别独立 Tabulate，每类只出现一次：
- `通用伤害 INC (Lightning,Cold,Fire) = 108%` — Damage mod，7 个来源
- `元素伤害 INC (Lightning,Cold,Fire) = 101%` — ElementalDamage mod，7 个来源
- `冰霜伤害 INC (Cold) = 10%` — ColdDamage mod，1 个来源
- `通用伤害 MORE = ×0.70` — 只出现一次

Formula items 从 15 → 13 个。

### Item 部位标注

因为 mod source 中 itemId 恒为 -1（`Item.lua:1760` 中 `self.id or -1`），无法通过 ID 匹配 slot。改用物品名称匹配 `orderedSlots`。

结果：
- `Rapture Shard, Sapphire (Jewel)` — 明确是珠宝
- `Damnation Grip, Unset Ring (Ring 2)` — 戒指
- `Adonia's Ego, Siphoning Wand (Weapon 1)` — 武器

### 新增功能

`full_analysis(skill_name="ball lightning")` — 按技能名称模糊匹配指定分析目标。

- `_find_socket_group_by_skill_name(lua, calcs, skill_name)`: 遍历所有 socketGroup，initEnv+perform 获取每组的主技能名+DPS，与用户输入做大小写不敏感的精确/部分匹配

### Lucky Hits 模拟分析

代码审查确认：
- Lua 端检查 `LuckyHits`、`ElementalLuckHits`、`CritLucky` 三个 Flag → 100%
- 否则 `Sum("BASE", cfg, dt.."LuckyHitsChance", "LuckyHitsChance")`
- **CritLucky 单独影响暴击判定**（roll 两次取高），与 Lucky Hits 独立
- The Spring Hare 提供 `LuckyHitsChance`（概率型），不是 Flag 型的全局 Lucky
- DPS breakdown 只做来源展示，实际 DPS 增益在 POB CalcOffence 的 Lucky 公式中计算
