**3. PassiveTree.lua L400-477 ProcessStats() 解析**：
```lua
function PassiveTreeClass:ProcessStats(node, startIndex)
    -- 初始化 modList
    node.modList = new("ModList")
    
    -- 逐行解析
    for i, line in ipairs(node.sd) do
        local list, extra = modLib.parseMod(line)  -- ← 核心：调用 ModParser
        
        if not list then
            node.unknown = true    -- 完全无法识别
        elseif extra then
            node.extra = true      -- 识别但无法完全理解
        end
        
        node.mods[i] = { list = list, extra = extra }
    end
    
    -- 构建 modList（仅包含完全识别的）
    for i, mod in ipairs(node.mods) do
        if mod.list and not mod.extra then  -- ← 关键：extra 非nil则跳过
            node.modList:AddMod(mod)
        end
    end
end
```

#### Phase C: ModParser.lua 文本解析引擎（6900+ 行）

**核心解析流程**：

1. **preFlagList scan**: 检查文本开头的 skill type/condition 标签
   ```lua
   -- L1290
   ["^meta skills [hd][ae][va][el] "] = { tag = { type = "SkillType", skillType = SkillType.Meta } }
   ```
   - `"Meta Skills gain 15% increased Energy"` → preFlag = `{ tag = { type = "SkillType", skillType = SkillType.Meta } }`
   - line 剩余: `"15% increased Energy "`

2. **formList scan**: 匹配修饰符形式
   ```lua
   -- L63
   ["^(%d+)%% increased"] = "INC"
   ```
   - `"15% increased"` → modForm = "INC", formCap = {"15"}
   - line 剩余: `" Energy "`

3. **modNameList scan**: 查找修饰符名称
   ```lua
   -- L155-895 定义了 ~700 个 modifier name 映射
   ["energy shield"] = "EnergyShield"
   ["maximum energy shield"] = "EnergyShield"
   ["energy shield recharge rate"] = "EnergyShieldRecharge"
   -- ❌ 但没有 ["energy"] = "MetaSkillEnergy" 这样的条目！
   ```
   - `"energy"` → **modName = nil**

4. **失败处理**：
   ```lua
   -- L6440-6441
   if not modName then
       return { }, line  -- 返回空列表 + 未解析的剩余文本
   end
   ```

### "Meta Skills gain 15% increased Energy" 解析结果

| 阶段 | 结果 |
|------|------|
| preFlagList | ✅ 成功识别 `SkillType.Meta` tag |
| formList | ✅ 成功识别 `INC` form，value=15 |
| modNameList | ❌ 找不到 "energy" → **modName = nil** |

### 数据可信度评估

#### ✅ 可信的部分

1. **天赋树原始文本是权威游戏数据**
   - `tree.lua` 中的 `stats` 字段来自游戏官方数据
   - `"Meta Skills gain 15% increased Energy"` 是精确的游戏内描述

2. **StatDescriptions 提供精确语义**
   - `stat_descriptions.lua` L132727-132835 包含完整能量 stat 定义
   - `"Meta Skills gain {0}% increased Energy"` → `energy_generated_+%`
   - `"Meta Skills gain {0}% more Energy"` → `ascendancy_energy_generated_+%_final`
   - 这些是**官方stat ID**，可通过 ModCache.lua 追溯到计算逻辑

3. **v4 提取器的分类是正确的**
   - 从描述文本识别出 INC/MORE/条件/特殊类型
   - 映射到官方 stat ID
   - 标记 `pob_status='unimplemented'` 是准确的

#### ❌ POB 不可计算的部分

1. **ModParser.lua 缺少能量 stat 映射**
   - 没有 `["energy"]` modifier name
   - 没有 `["energy generation"]` modifier name
   - 没有 `["energy generated"]` modifier name

2. **CalcSetup/CalcTriggers 缺少能量计算逻辑**
   - 已知 `calcs.triggers(env, env.player)` 被注释掉（CalcPerform.lua L3090）
   - SkillStatMap.lua 没有能量 stat → modifier 映射
   - 没有代码处理能量积累、触发阈值、能量成本

3. **天赋节点的能量修饰符被静默丢弃**
   - `node.extra = true` → 不加入 `modList`
   - CalcSetup 读取 `modList` → 得到空结果
   - 计算结果**不包含能量加成**

## 结论

**用户问题的答案**：

1. **天赋点描述文本如何生效**：
   - 通过 ModParser.lua 文本→modifier 解析管道
   - 解析结果存储在 `node.modList` 中
   - CalcSetup 在计算阶段读取 `modList` 并应用

2. **数据是否可信**：
   - **✅ 数据层可信**：天赋文本是权威游戏数据，StatDescriptions 提供精确语义
   - **❌ 计算层不可信**：POB 目前无法计算能量系统，修饰符被静默丢弃
   - **✅ 我们的提取方法正确**：识别、分类、映射都是准确的

3. **为什么 POB 不生效**：
   - ModParser.lua 的 `modNameList` 中没有 "energy" 条目
   - 导致 `modName = nil`，返回空 modList
   - PassiveTree 标记 `node.extra = true`，跳过该修饰符

这是一个**游戏已实现但 POB 未实现**的典型案例。

## 验证：游戏已实现 vs POB 未实现（2026-03-10 续）

### 验证问题

用户提出两个验证问题：
1. **是否确实是游戏已实现但 POB 未实现？**
2. **实体库的天赋树提取是否有问题？**

### ModCache.lua 证据

**关键发现**：ModCache.lua 是 POB 预运行 ModParser.lua 对所有已知 mod 文本解析后的**缓存结果**。

```lua
-- ❌ 无法解析（值为 nil）
c["Meta Skills gain 15% increased Energy"]={nil,"Meta Skills gain 15% increased Energy "}
c["Meta Skills gain 35% more Energy"]={nil,"Meta Skills gain 35% more Energy "}

-- ✅ 成功解析（有 modifier 定义）
c["Meta Skills have 20% increased Reservation Efficiency"]={{[1]={...name="ReservationEfficiency",type="INC",value=20}},nil}

-- ⚠️ 解析为空表（特殊标志）
c["Energy Generation is doubled"]={{},"Energy Generation  "}
```

**结论**：ModCache.lua 直接证明了 POB **无法解析** Meta Energy 修饰符（`{nil, ...}` 表示解析失败）。

### 实体库提取验证

**查询结果**：

| Invocation Energy 相关 | 3 | ✅ 正确提取 |
| "Energy Generation is doubled" | 0 | ✅ 正确（tree.lua 中无此节点）|

**详细数据**：

### "Energy Generation is doubled" 来源调查

**关键发现**：`energy_generation_is_doubled` stat 存在于 StatDescriptions (#6015)，但**不存在于任何 POB 数据源**：

- ❌ tree.lua（天赋树）- 无此节点
- ❌ Skills/*.lua（技能文件）- 无此 stat
- ❌ Uniques/*.lua（唯一物品）- 无此 stat
- ❌ Bases/*.lua（物品基础）- 无此 stat

**结论**：这个 stat 可能是游戏未来更新的内容、测试/开发中的 stat、或某个特殊机制的效果。POB 数据中**根本没有来源**，因此实体库没有遗漏。

tree.lua 中 passive_63236 (The Soul Springs Eternal) 的图标文件名是 `InvokerEnergyDoubled.dds`，但节点实际文本是：
- "Meta Skills gain 35% more Energy"

### 最终结论

1. **✅ 确认：游戏已实现但 POB 未实现**
   - ModCache.lua 的 `{nil, ...}` 值是直接证据
   - ModParser.lua 的 `modNameList` 缺少 "energy" 条目

2. **✅ 实体库提取完全正确**

   - `stats_node` 字段存储的是描述文本数组
   - 没有遗漏或错误

3. **✅ v4 提取器的分类准确**
   - INC/MORE/条件/特殊分类与 StatDescriptions 一致

**示例**：
- 玩家有珠宝 "Meta Skills gain 8% increased Energy"
- 公式库不会计入这 8% INC
- 计算结果会比实际低 8%
