## Doedre 绕过机制正确因果链（代码验证完成）

### 核心发现
Doedre's Undoing 绕过 Triggered 约束的真实原因**不是 Hazard 标签替换**，而是：

1. **Meta 技能通过隐藏 Support（SupportMeta*）给子技能添加 Triggered 标签**
   - 每个 Meta 技能宝石都有 `additionalGrantedEffectId1 = "SupportMeta*Player"`
   - SupportMeta*.addSkillTypes = { SkillType.Triggered }
   - SupportMeta*.excludeSkillTypes = { SkillType.InbuiltTrigger }
   - 在 CalcActiveSkill.lua:129-137 的 createActiveSkill Pass 1 中添加

2. **Doedre 的隐藏爆炸技能自带 InbuiltTrigger 标签**
   - TriggeredCurseZoneHazardExplosionPlayer.skillTypes 包含 InbuiltTrigger
   - 所有 SupportMeta* 的 excludeSkillTypes 都包含 InbuiltTrigger
   - 因此 Meta 的隐藏 Support 无法辅助爆炸技能 → 不会给它添加额外约束

3. **结果对比**
   - 诅咒技能：被 SupportMeta* 辅助 → 添加 Triggered → 受所有 Triggered 约束
   - 爆炸技能：被 SupportMeta* 排除 → 不被额外约束 → 能独立造成伤害

### 之前分析的错误
- 错误1：认为 Hazard 标签本身是绕过原因（实际上 Hazard 只让诅咒变成危险区域）
- 错误2：没有追踪 Meta 隐藏 Support 的 addSkillTypes 传播路径
- 正确路径：Meta SupportMeta* → excludeSkillTypes={InbuiltTrigger} → 爆炸技能被排除

### 对框架改进的启示
通用绕过检测需要追踪：
1. Support 的 excludeSkillTypes 排除了哪些技能
2. additionalGrantedEffectId 关联的隐藏技能有什么标签
3. 排除 = 绕过约束的核心机制（不是添加标签，而是**不被添加约束**）

## 探索模式：Meta 自循环限制深度分析

### 核心发现：CalcTriggers.lua 不建模 Meta 能量系统

1. **CalcTriggers.lua 中不存在**：centienergy、Meta、SupportMeta、Doedre、generic_ongoing_trigger
2. `calcs.triggers()` 入口（行1436-1465）通过 configTable 查找触发配置，处理的是 POB DPS 计算
3. `isTriggered()` 函数（行40-42）用于 4 处 `not isTriggered(skill)` 过滤（行230,396,592,1043），阻止被触发技能作为 DPS 触发源
4. Meta 能量 stat（如 `cast_on_crit_gain_X_centienergy_per_monster_power_on_crit`）在 SkillStatMap.lua 中**未映射**（仅有 `generic_ongoing_triggerer_is_invocation_skill` → Condition:InvocationSkill）

### isTriggered() vs Meta 能量系统

- `isTriggered()` = POB 的 DPS 计算约束（被触发技能不能作为触发源）
- Meta 能量积累 = 游戏引擎 stat 系统（centienergy），POB 不建模
- 两者是**不同层面**的机制

### 自循环分析路径

**已确定**：
- SupportMeta* 排除 InbuiltTrigger → 不辅助 Doedre 爆炸 → 爆炸不在 Meta 管道中
- configTable["cast on critical strike"] 要求触发源是 Attack → Spell 爆炸不能作为源

**待确定**：
- 游戏引擎中爆炸造伤是否为 Meta 积累 centienergy（POB 不建模此机制）
- centienergy stat 的来源限制（是否只来自主动使用的技能）

### 关键修正：爆炸伤害独立 — 两个并行触发系统

**核心铁证**：
- `TriggeredCurseZoneHazardExplosionPlayer` 的 stat 是 `triggered_by_curse_zones_are_hazards_%`（不是 `triggered_by_generic_ongoing_trigger`）
- 爆炸技能有 InbuiltTrigger → 被所有 SupportMeta* 排除 → 不在 Meta effectList 中
- 爆炸不带 `triggered_by_generic_ongoing_trigger` → 不被 Meta 能量系统标记

**完整因果链**：
1. Meta 系统：玩家行为 → centienergy 积累 → 触发诅咒（被 SupportMeta* 辅助，带 Triggered 标签）
2. Doedre 系统：诅咒被触发 → `curse_apply_as_curse_zone` → 区域变成 Hazard → 敌人进入 → `triggered_by_curse_zones_are_hazards_%` → 独立爆炸伤害

**结论**：不是"绕过自循环限制"，而是**两个独立触发系统并行运作**。爆炸伤害从来就不在 Meta 管道内，是平行的免费额外伤害。

**Doedre 的特殊性**：大多数 Support 不产生 additionalGrantedEffect。Doedre 产生了一个带 InbuiltTrigger 的独立技能，天然逃逸 Meta 控制范围。

**对 bypass 框架的启示**：真正的"绕过"不是标签排除，而是**并行触发管道的创建**。Support 通过 additionalGrantedEffect 创建独立技能，该技能的 InbuiltTrigger 使其不受 Meta 约束。

## 通用约束绕过方法论（从 Doedre 案例推广）

### 核心模型：三层递进判定

```
Layer 1：管道识别 — 约束存在于哪条效果管道中？
Layer 2：路径分叉 — 是否存在分叉点创建了并行管道？
Layer 3：效果等价 — 并行管道能否产生等价或互补效果？
```

三层全满足 → 真正绕过。当前框架只做了 Layer 1 的部分。

### Layer 2 路径分叉的数据特征（通用）

| 模式 | 数据特征 | 创建并行管道的原因 |
|------|---------|-------------------|
| additionalGrantedEffect | 宝石有 additionalGrantedEffectId 指向独立主动技能 | 独立 activeSkill 有自己的 skillTypes |
| InbuiltTrigger | 技能自带 InbuiltTrigger+Triggered | 不受 Meta Support 管道控制 |
| 独立触发 stat | 有 triggered_by_X（非 generic_ongoing_trigger） | 触发路径独立于 Meta 能量系统 |
| SkillGrantedBySupport | 隐藏技能由 Support 授予 | 不受 Support 约束链控制 |
| fromItem/fromTree | 效果来源非插槽 | 绕过插槽级辅助链 |

### Layer 3 效果等价判定

| 效果类型 | 判定条件 | 绕过价值 |
|----------|---------|---------|
| 造伤 | Damage标签 + base damage stat | 高 |
| 施加状态 | apply_*_on_hit 类 stat | 中 |
