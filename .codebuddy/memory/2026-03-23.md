## pob-build-analyzer v1.0.3 — 天赋节点多行 mod 解析修复

### 用户疑问验证

#### 疑问1：CritMultiplier +50 BASE 为什么给 +23.4% DPS

**用户质疑正确！之前的 What-If 分析使用了错误的 mod 类型。**

POB CritMultiplier 公式（CalcOffence.lua:3473-3476）：
```
extraDamage = (BASE / 100) × (1 + INC / 100) × MORE
output.CritMultiplier = 1 + max(0, extraDamage)
```

该构筑的实际 modDB 数据：BASE=100, INC=200, MORE=1.0 → extraDamage=3.0 → CritMultiplier=4.0

游戏词缀对应关系：
- "+X% to Critical Damage Bonus" → CritMultiplier **BASE** X （稀有词缀，会被 INC 放大）
- "X% increased Critical Damage Bonus" → CritMultiplier **INC** X （常见词缀，线性叠加）

**之前 What-If 错误地用了 BASE 50**（被 200% INC 放大 3 倍 → CritMultiplier 4.0→5.5），应该用 **INC 50**（线性叠加 → CritMultiplier 4.0→4.5）。

| 注入方式 | CritMultiplier | DPS 变化 | 适用场景 |
|----------|----------------|----------|----------|
| BASE 50（错误使用）| 4.0→5.5 | +23.4% | "+50% to Critical Damage Bonus" |
| **INC 50（正确）** | 4.0→4.5 | **+7.8%** | "50% increased Critical Damage Bonus" |

**影响**：之前 What-If 灵敏度分析中 CritMultiplier 排名第一（+23.4%）是错误的，实际应为 +7.8%，排名需重新评估。

#### 疑问2：Spark 是否属于 Storm Spell

**结论：Spark 不是 Storm Spell ❌ → 发现并修复了 Bug**

- Spark skillTypes: Spell(2), Projectile(3), Damage(7), Duration(9), Lightning(30), Triggerable(31) 等 14 个
- **没有 Storm(255) 也没有 Plant(252)**
- SkillType.Storm = 255, SkillType.Plant = 252（定义在 Global.lua:592-595）

### Bug 根因

`build_loader.py` 缺少 POB 桌面版 `ProcessStats` 的**多行 stat 合并逻辑**：

1. Wildsurge Incantation 有 4 行 stat：`"Storm and Plant Spells:"` / `"deal 50% more damage"` / `"cost 50% less"` / `"have 75% less duration"`
2. POB 桌面版：单行解析失败后，合并 4 行为 `"Storm and Plant Spells: deal 50% more damage cost 50% less have 75% less duration"`，ModParser 匹配成功 → 生成 3 个带 `{ type = "SkillType", skillTypeList = { SkillType.Storm, SkillType.Plant } }` tag 的 mod
3. 我们的 build_loader：无合并逻辑 → `"Storm and Plant Spells:"` 被跳过 → `"deal 50% more damage"` 单独解析为无条件 `Damage MORE 50`

### 修复

在 `build_loader.py` 的两处天赋节点 stat 解析位置，完整移植 POB 桌面版 ProcessStats 的多行合并逻辑。

### 验证

- Spark + Wildsurge: DPS 从修复前 +49.9% → 修复后 0%（正确）
- Wildsurge modList: 1 个无条件 mod → 3 个带 SkillType tag 的 mod
- 精度不变: 22/24 (92%)
- **此 Bug 意味着之前 What-If 天赋探索中涉及多行 stat 的 Keystone 结果可能不准确**

### 用户提出的三个问题（续）

#### 问题1 续：CritMultiplier 公式澄清

用户问 "BASE=100 → CritMulti=4.0 怎么算出来的"

**POB 公式**（CalcOffence.lua:3473-3476）：
```
extraDamage = Sum(BASE)/100 × (1 + Sum(INC)/100) × More
CritMultiplier = 1 + max(0, extraDamage)
```

POE2 默认 `BASE=100`（来自 `Misc.lua` 的 `characterConstants.base_critical_hit_damage_bonus = 100`），
在 `CalcSetup.lua:615` 注入到 modDB：
```lua
modDB:NewMod("CritMultiplier", "BASE", data.characterConstants["base_critical_hit_damage_bonus"], "Base")
```

该构筑 BASE=100, INC=200：
- extraDamage = (100/100) × (1 + 200/100) = 1 × 3 = 3.0
- CritMultiplier = 1 + 3.0 = **4.0**

注意：INC 是对 BASE 的**乘法放大**，不是简单加法。但在 BASE=100 这个特殊值下，
`(100/100) × (1+INC/100)` 简化为 `(100+INC)/100` — 用户说的 "(100+INC) = CritMulti×100" 恰好成立。

INC +50 → CritMulti = 1 + (100/100)×(1+250/100) = 1 + 3.5 = **4.5**（+12.5%）
DPS 经 CritEffect 稀释：+12.5% × 29.3% / (1-0.293+0.293×4.0) = **+7.8%**

## _detect_is_spell() Bug 修复

### 问题

`_detect_is_spell()` 使用 `env.player.mainSkill.skillFlags.spell` 判断法术类型，但 Spark 构筑中 `skillFlags` 为 nil，导致法术检测失败，flat damage 攻击专属 profile 未被排除。

### 根因

`skillFlags` 表在 POB 的 `CalcActiveSkill.lua` 中由 `buildSkillFlags()` 设置，但在某些构筑/技能组合下可能为 nil（可能与 initEnv 重建 env 的时机有关）。而 `skillCfg.flags` 是始终可用的位域值。

### 修复

改用 `ms.skillCfg.flags` 位运算：`flags & 0x02 ~= 0` 检查 `ModFlag.Spell`（Global.lua:220）。

### 验证

## pob-build-analyzer v1.0.6 — 等基准灵敏度 + INC 合并修复

### 用户提出的三个问题

1. **INC 分母不一致**：Damage INC=36%、ElementalDamage INC=101%、LightningDamage INC=0% — 但 POB 中它们是同一乘区
2. **英文 label 被删除**：添加中文时误删了英文说明列
3. **对比不公平**：不同维度注入不同值，DPS% 变化无法横向比较

### INC 合并机制（CalcOffence.lua 根因分析）

**CalcOffence.lua:53-63**：`damageStatsForTypes` magic table 根据 `typeFlags` 位运算生成合并 mod 名称：
- Lightning(0x02) → `{"Damage", "LightningDamage", "ElementalDamage"}`
- Fire(0x08) → `{"Damage", "FireDamage", "ElementalDamage"}`
- Physical(0x01) → `{"Damage", "PhysicalDamage"}`

**CalcOffence.lua:135-136**：`inc = 1 + skillModList:Sum("INC", cfg, unpack(modNames)) / 100`
所有相关 stat 的 INC 合并为单一值。

**修复**：`_query_all_merged_inc()` 在 Lua 端复现此合并逻辑，自动检测主伤害类型，返回合并 INC 值。

### 等基准对比设计

- 旧方案：注入固定值（+50 INC、+20 MORE、+2 BASE），不同维度不可比
- 新方案：固定 DPS +30% 目标，二分搜索每个维度所需注入值
- `needed_value` 越小 = 性价比越高 = 优化杠杆越大
- 无法达到目标（如穿透在敌人负抗时）返回 None
